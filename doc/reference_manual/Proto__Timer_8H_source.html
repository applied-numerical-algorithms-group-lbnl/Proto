21 /** TraceTimer class is a self-tracing code instrumentation system
22 
23  TraceTimer class is a self-tracing code instrumentation system
24  for Chombo (or any other package really). The user interface is specified
25  by a small set of macros. The usage model is that you just leave these
26  timers in the code, for good. Initially, your application will have 'main'
27  and a few hewavy functions instrumented, and the lower level Chombo library
28  instrumentation. As your tool or application matures, it
29  will garner a larger set of instrumentation giving clear views of your
30  code performance. After a routine has been
31  cleverly and lovingly optimized, you leave in the timers, to spot when
32  some later bug fix or *improvement* undoes your previous labors.
33 
34  \note
35  You should never need to use or interact with the the classes TraceTimer or
36  AutoStart. Use the macros. They call the right functions and classes for you.
37 
38  The first macro is what people will use the most:
39  \code
40  PR_TIME("label");
41  \endcode
42 
43  This is the simplest interface for timers. you place this macro call in a function
44  you wish to be timed. It handles making the timer, calling 'start' when you
45  enter the function, and calling 'stop' when you leave the function. A good
46  idea is to use a 'label' specific enough to be unambiguous without being
47  overwhelming. for instance:
48 
49  \code
50  void AMRLevelPolytropicGas::define(AMRLevel* a_coarserLevelPtr,
51  const ProblemDomain& a_problemDomain,
52  int a_level,
53  int a_refRatio)
54  {
55  PR_TIME("AMRLevelPolytropicGas::define");
56  .
57  .
58  }
59  \endcode
60 
61  In this case, we have a class with many constructors and define functions that
62  all funnel into a single general function. We can just call this 'define' and
63  not worry about naming/instrumenting all the different overloaded instances. If
64  you slip up and use the same label twice, that is not a real problem, the two
65  locations will be timed and tracked properly (even if one is a sibling or parent
66  of the other). The only place it will make things a little harder is in the output
67  where you might have the same name show up and look confusing.
68  <br><br>
69  In serial, you will see a file called <em>time.table</em> (in parallel, you will get a
70  <em>time.table.n</em> (where n is the rank number) files). If
71  you want fewer files, you can do
72  setenv PR_OUTPUT_INTERVAL nproc
73  and it will only output every nproc processors time.table.n files
74  (where n%nproc == 0). I won't go into this file
75  format. It is kind of gprof-ish, with what I consider improvements. The real
76  benefit here is profiling that understands our Chombo context, a smaller information
77  set to observe, and the fact that, so far in my testing, the timers have negligible
78  impact on the run time or memory use of the code.
79  <br><br>
80  By default, Chombo compiles in the instructions for the timers wherever the macros
81  appear. If the compiler macro <b>PR_NTIMER</b> is defined, then all the PR_TIME* macros
82  evaluate to empty expressions at compile time.
83 
84  \par So, you put some PR_TIME calls in your code and ran it, and nothing happened:
85  Chombo looks for the environment variable <b>PR_TIMER</b>. If it is set to anything (even
86  if it is set to 'false' or 'no' or whatever) then the timers will be active and
87  reporting will happen. If this environment variable is not set, then all the timers
88  check a bool and return after doing nothing.
89  \par
90  One point of interest with using the environment variable: In parallel jobs using
91  mpich, only processor 0 inherits the environment variables from the shell where
92  you invoke 'mpirun', the rest read your .cshrc (.bashrc, etc.) file to get their
93  environment. To time all your processes, you need to make sure the <b>PR_TIMER</b>
94  environment variable gets to all your processes.
95 
96 
97 
98  \par Auto hierarchy:
99  The timers automatically figure out their parent/child relationships. They
100  also can be placed in template code. This has some consequences. First,
101  if you have a low level function instrumented that has no timers near it in
102  the code call stack, you will see it show up as a child of a high level timer.
103  the root timer "main" will catch all orphaned timers. So, even though you
104  might make no call to, say, 'exchange' in your 'main' function, you might
105  very well call a function, that calls a function, that calls 'exchange'. Since
106  no code in between was instrumented, this exchange is accounted for at 'main'.
107  This might look strange, but it should prove very powerful. An expensive orphan
108  is exactly where you should consider some more timers, or reconsidering code
109  design.
110 
111  \par
112  For performance reasons, child timers have only one parent. As a consequence
113  each PR_TIME("label") label can show up at multiple places in your output. Each
114  instance has it's own timer. So, each path through the call graph that arrives
115  at a low-level function has a unique lineage, with it's own counter and time.
116  Thus, I can instrument LevelData::copyTo once, but copyTo can appear in many
117  places in the time.table file.
118 
119 
120  The next level up in complexity is the set of *four* macros for when you want
121  sub-function resolution in your timers. For instance, in a really huge function
122  that you have not figured out how to re-factor, or built with lots of bad cut n paste
123  code 're-use'.
124  \code
125  PR_TIMERS("parent");
126  PR_TIMER("child1", t1);
127  PR_TIMER("child2", t2);
128  PR_START(t1);
129  PR_STOP(t1);
130  PR_START(t2);
131  PR_STOP(t2);
132  PR_START(t1);
133  PR_STOP(t1);
134  \endcode
135 
136  PR_TIMERS has the same semantic as PR_TIME, except that you can declare an
137  arbitrary number of children after it in the same function scope. The
138  children here do not autostart and autostop, you have to tell them where to
139  start and stop timing. The children can themselves be parents for timers
140  in called functions, of course. The children obey a set of mutual exclusions. The
141  following generate run time errors:
142  - double start called
143  - double stop called
144  - start called when another child is also started
145  - you leave the function with a child not stopped
146 
147  the following will generate compile time errors:
148  - more than one PR_TIME macro in a function
149  - invoking PR_TIMER("child", t) without having first invoked PR_TIMERS
150  - re-using the timer handle ie. PR_TIMER("bobby", t1); PR_TIMER("sally", t1)
151  - mixing PR_TIME macro with PR_TIMER
152  - mixing PR_TIME macro with PR_TIMERS
153 
154  You do not have to put any calls in your main routine to activate the clocks
155  or generate a report at completion, this is handled with static iniitalization
156  and an atexit function.
157  <br><br>
158  There is a larger argument of manual instrumentation being counter to good development.
159  Profiling the code is supposed to tell you where to expend your optimization effort.
160  Manual instrumentation opens the door to people wasting time *assuming* what parts of the
161  code are going to take up lots of time and instrumenting them, before seeing any real
162  performance data. Good judgement is needed. We have a body of knowledge about Chombo
163  that will inform us about a good minimal first set of functions to instrument.
164 */
